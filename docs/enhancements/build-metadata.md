# Build Metadata Display

## Goal

Show a subtle build metadata line inside the hero intro profile panel, directly above `Contact & Links`.

Final display format:

```text
Last Build: DD MMM YYYY HH:MM AM/PM (COMMIT_ID)
```

Example local preview:

```text
Last Build: 09 May 2026 11:36 AM (4306f4f)
```

The datetime links to the GitHub Actions workflow run. The short commit ID links to the GitHub commit.

## Current Implementation

The site uses runtime JSON metadata instead of mutating `dist/index.html` after the Vite build.

The GitHub Actions workflow writes `site/public/data/deployment.json` at workflow runtime. During app deploys, Vite copies that file into `site/dist/data/deployment.json`; during data-only deploys, the workflow syncs `site/public/data` directly. The React app loads the file with the other `/data/*.json` content and renders the build line in `IntroTerminal`.

This keeps the build deterministic and lets the UI handle formatting, links, accessibility, light/dark themes, and mobile wrapping through normal React and CSS.

## Files

- `.github/workflows/deploy.yml` generates deployment metadata during each workflow run.
- `site/public/data/deployment.json` is committed as a local preview sample and overwritten by GitHub Actions during deploys.
- `site/src/lib/data.js` loads `/data/deployment.json` as optional data.
- `site/src/App.jsx` passes `deployment` into `IntroTerminal`.
- `site/src/components/IntroTerminal.jsx` formats and renders the build line.
- `site/src/styles.css` styles the line as subtle profile-panel metadata.

## Metadata JSON

The JSON can keep a richer shape for debugging and future use, but the current UI only consumes:

- `deployedAt` - workflow runtime timestamp generated by the runner.
- `runUrl` - GitHub Actions run URL used by the datetime link.
- `shortSha` or `sha` - displayed commit ID, preferring `shortSha` and falling back to the first 7 characters of `sha`.
- `commitUrl` - GitHub commit URL used by the commit ID link.

Committed local preview sample:

```json
{
  "environment": "production",
  "deployedAt": "2026-05-09T03:36:32Z",
  "runId": "25590639374",
  "runNumber": "7",
  "runAttempt": "1",
  "runUrl": "https://github.com/teamcmcbot/zhenwei-dev-site/actions/runs/25590639374",
  "sha": "4306f4fa7bf610bccf401338d7c4237401566ce1",
  "shortSha": "4306f4f",
  "commitUrl": "https://github.com/teamcmcbot/zhenwei-dev-site/commit/4306f4fa7bf610bccf401338d7c4237401566ce1",
  "ref": "main",
  "actor": "teamcmcbot",
  "deployType": "app",
  "status": "completed",
  "conclusion": "success"
}
```

The extra fields are intentionally kept in the data file for context, but they are not referenced by the current JSX or CSS.

## Workflow Generation

`deployedAt` is generated when the GitHub Actions workflow runs. It is not the commit timestamp.

Current generation pattern:

```yaml
- name: Write deployment metadata
  run: |
    mkdir -p site/public/data
    DEPLOYED_AT="$(date -u +%Y-%m-%dT%H:%M:%SZ)"
    SHORT_SHA="${GITHUB_SHA::7}"
    RUN_URL="${GITHUB_SERVER_URL}/${GITHUB_REPOSITORY}/actions/runs/${GITHUB_RUN_ID}"
    COMMIT_URL="${GITHUB_SERVER_URL}/${GITHUB_REPOSITORY}/commit/${GITHUB_SHA}"

    cat > site/public/data/deployment.json <<EOF
    {
      "environment": "production",
      "deployedAt": "${DEPLOYED_AT}",
      "runId": "${GITHUB_RUN_ID}",
      "runNumber": "${GITHUB_RUN_NUMBER}",
      "runAttempt": "${GITHUB_RUN_ATTEMPT}",
      "runUrl": "${RUN_URL}",
      "sha": "${GITHUB_SHA}",
      "shortSha": "${SHORT_SHA}",
      "commitUrl": "${COMMIT_URL}",
      "ref": "${GITHUB_REF_NAME}",
      "actor": "${GITHUB_ACTOR}",
      "deployType": "${{ inputs.deploy_type }}"
    }
    EOF
```

The step must run before `npm run build` for app deploys so Vite copies the generated file into `dist/data`.

## Local Development

The chosen local strategy is to commit a sample `site/public/data/deployment.json` populated from a known successful workflow run.

Benefits:

- Local `npm run dev` shows the same UI shape as production.
- The data schema is visible in the repository.
- There is no local 404 for `/data/deployment.json`.
- GitHub Actions overwrites the file with current metadata during deployment.

`site/src/lib/data.js` still treats the metadata file as optional so the page will not fail if the file is missing or temporarily invalid.

## Rendering

`IntroTerminal` renders the metadata as one inline sentence:

```jsx
<div className="intro-build-metadata" aria-label="Build metadata">
  <p>
    <span>Last Build:</span>{" "}
    <a href={deployment.runUrl} target="_blank" rel="noopener noreferrer">
      <time dateTime={deployment.deployedAt}>{formattedDeployTime}</time>
    </a>
    {" ("}
    <a href={deployment.commitUrl} target="_blank" rel="noopener noreferrer">
      {shortSha}
    </a>
    )
  </p>
</div>
```

Fallback behavior:

- Missing `deployedAt` displays `Local preview`.
- Invalid `deployedAt` displays the original string.
- Missing `runUrl` leaves the datetime unlinked.
- Missing `shortSha` falls back to `sha.slice(0, 7)`.
- Missing `commitUrl` leaves the commit ID unlinked.

Date formatting:

- Format: `DD MMM YYYY HH:MM AM/PM`.
- The workflow stores `deployedAt` in UTC/GMT using `date -u`.
- The UI displays that timestamp in Singapore time with `timeZone: "Asia/Singapore"`.
- Example: `09 May 2026 11:36 AM`.

## Styling

The build line is intentionally subtle and footer-like. It uses existing theme variables rather than hard-coded colors.

Key styling behavior:

- `margin-top: auto` on `.intro-build-metadata` pushes the build line and `Contact & Links` group toward the bottom of the intro profile panel.
- `.intro-profile-contact` no longer owns the auto margin, so there is no large gap between the build line and contact links.
- The line uses a small monospace font and muted text color.
- The label uses `--terminal-cyan`.
- Links inherit the muted color and only turn `--terminal-green` on hover/focus.
- Long values use `overflow-wrap: anywhere` so mobile layouts do not overflow.

Current CSS shape:

```css
.intro-build-metadata {
  margin-top: auto;
  min-width: 0;
  padding-top: 0.75rem;
  border-top: 1px solid var(--line);
}

.intro-build-metadata p {
  margin: 0;
  min-width: 0;
  font-family: "SFMono-Regular", Consolas, "Liberation Mono", monospace;
  color: var(--muted);
  font-size: 0.76rem;
  line-height: 1.4;
}

.intro-build-metadata p > * {
  min-width: 0;
  overflow-wrap: anywhere;
}

.intro-build-metadata span:first-child {
  color: var(--terminal-cyan);
  letter-spacing: 0;
}

.intro-build-metadata a {
  color: var(--muted);
  text-decoration: none;
}

.intro-build-metadata a:hover,
.intro-build-metadata a:focus-visible {
  color: var(--terminal-green);
  text-decoration: underline;
}
```

## Accessibility

- The wrapper uses `aria-label="Build metadata"`.
- The timestamp uses a semantic `time` element with `dateTime` set to the ISO timestamp when available.
- Both links use `target="_blank"` and `rel="noopener noreferrer"`.
- Existing global `:focus-visible` styles provide keyboard focus indication.

## Caching And Deployment Behavior

The existing deployment workflow already applies no-cache headers to JSON data:

- App deploy: syncs `site/dist/data` with `public,max-age=0,must-revalidate`.
- Data-only deploy: syncs `site/public/data` with `public,max-age=0,must-revalidate`.

This is appropriate for `deployment.json` because it should reflect the latest workflow run quickly.

CloudFront invalidation already covers the file:

- App deploys invalidate `/*`.
- Data-only deploys invalidate `/data/*`.
